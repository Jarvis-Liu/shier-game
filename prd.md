📝 猜数字 PvP 联机对战游戏 (NumDecode) 需求文档
1. 项目概述
项目名称：NumDecode (数字破译)
游戏类型：逻辑博弈、休闲联机、二次元手绘风格
核心玩法：双方玩家各自设定一个 4 位秘密数字（支持重复），轮流猜测对方的数字。系统仅提示位置和数字完全正确的个数（例如：3个对）。先猜中者获胜。
视觉风格：手绘水彩卡通 + 复古纸张纹理 + 铅笔线条。
2. 核心页面逻辑 (Frontend)
项目采用 Nuxt 3 动态路由，主要分为以下两个核心页面：
2.1 游戏大厅 (/)
用户登录：
输入框：填写玩家昵称（Identifier）。
本地存储：使用 localStorage 记忆昵称。
房间选择：
创建项目：点击生成随机 4 位房间码，跳转至房间页并设为 Host。
加入项目：输入指定 4 位房间码，校验成功后进入。
交互动效：按钮点击有水彩晕染扩散效果。
2.2 游戏房间 (/room/[id])
身份判定：
前两位进入者为 Player（对战方）。
后进入者自动转为 Spectator（观众）。
对局阶段控制：
准备阶段 (Setting)：双方各自输入自己的 4 位“秘密数字”。
对战阶段 (Playing)：轮流操作。自己的回合可提交猜测，对方回合等待。
结算阶段 (Finished)：显示胜负结果，支持点击“再来一局”。
信息展示：
战术日志 (Battle Log)：左侧显示历史记录。
操作台 (Console)：右侧为 0-9 数字键盘和提交按钮。
观众席：底部显示当前在线的观察者名单。
3. 服务端接口设计 (Backend API / Nitro)
由于使用 Nuxt 3，接口位于 /server/api 目录下。
3.1 房间管理 API
接口路径
方法
说明
/api/room/create

创建房间，生成唯一 ID，初始化房间状态。
/api/room/check
GET
检查房间是否存在、当前人数及对局状态。

3.2 WebSocket 事件 (Socket.io / H3)
后端需维护内存中的 rooms 对象（或 Redis），处理实时通讯：
客户端发送 (Emit)：
join_room: 携带昵称进入房间。
set_secret: 提交本局自己的秘密数字。
submit_guess: 提交猜测的 4 位数字。
服务端广播 (Broadcast)：
room_state: 同步当前房间内玩家状态、对局进度。
guess_result: 广播猜测结果（对猜方显示数字+结果，对另一方及观众仅显示结果个数）。
game_over: 宣布获胜者。
4. 视觉规范 (Visual Identity)
4.1 UI 元素定义
背景纹理：泛黄、带有细微纤维的旧纸张贴图。
边框设计：3px 宽的深石墨色铅笔线，线条不完全平直，有手抖动感。
色彩分配：
己方操作：水彩柔蓝 (rgba(56, 189, 248, 0.4)) 晕染背景。
对手操作：水彩暖橙 (rgba(251, 146, 60, 0.4)) 晕染背景。
成功反馈：草莓红或墨绿色水彩点缀。

5. 技术栈建议
框架：Nuxt 4
样式：Tailwind CSS (用于快速布局) + CSS 遮罩/滤镜 (实现水彩质感)
状态管理：Pinia (管理玩家信息与房间同步状态)

字体：中文字体推荐“ZCOOL 快乐体”或类似手写感开源字体
